# TraceCraft 规划书（架构调整版）

**版本**：V2.2
**日期**：2026-06-15
**名称**：TraceCraft（可选名：`TraceCraft`、`RunSketch`）
**状态**：V1.0 MVP 已入库（后台原型 + 前端 Demo 跑通）；产品策略已收窄为“城市创意跑路线 + 跑团挑战工具”，新增“路线预览 + 风险提示 + 用户确认”作为导航前置关口

---

## 模块1：产品核心定位

### 一句话定义
TraceCraft 是一款“**图片/形状转跑步路径并可实时导航的城市创意运动工具**”：用户上传任意图片或选择模板后，系统生成尽量安全可跑、适合分享和组织挑战的路线，并在 App 内直接导航，不出现 GPX 文件概念。

### 产品定位判断
- **不是高频刚需跑步工具**：不以替代 Keep、咕咚、Strava 等训练记录 App 为目标。
- **核心价值是创意和组织**：让跑步路线有形状、有故事、可传播、可发起挑战。
- **第一阶段优先服务小众高动机用户**：跑团、校园社团、内容创作者、品牌活动组织者，比泛城市跑者更容易形成种子用户。
- **留存关键不是“能生成图形”，而是“路线真的能跑”**：如果路线穿越封闭道路、河流、快速路或复杂立交，只能当玩具；只有接入真实路网约束后，才具备持续使用价值。

### 核心差异化优势（3点）
1. **画像式路线生成**：输入从“文字/Logo/表情包/涂鸦图”到轨迹，不需要先选路线。
2. **目标里程闭环控制**：输入 3km/5km/半马，后台按几何保形缩放（含节点数限幅、曲率平滑）直接产出可跑路径。
3. **地图 provider 与坐标体系统一适配**：路线引擎对坐标标准化，不受不同地图 SDK 影响，天然支持国内外切换（高德/Google 先行、百度/腾讯保留位）。

### 目标用户画像
1. **跑团 / 校园社团 / 健身工作室组织者（优先级 P0）**
   - 场景：发起“爱心跑”“校徽跑”“城市地标跑”“品牌 Logo 跑”等活动。
   - 痛点：普通路线缺少话题，活动组织者需要低门槛生成挑战路线、分享海报和打卡结果。
   - 付费可能：较高，适合按活动、模板包、团队功能收费。
2. **内容创作者 / 社交分享型跑者（优先级 P0）**
   - 场景：小红书、抖音、朋友圈发布路线作品，用运动轨迹做故事表达。
   - 痛点：不想处理 GPX、地图编辑、坐标转换等复杂操作。
   - 付费可能：中等，愿意为好看的模板、分享卡片、高清导出付费。
3. **城市年轻跑者（优先级 P1）**
   - 场景：晨练、夜跑、周末打卡；希望路线有新鲜感。
   - 痛点：常规路线重复，但对安全、红绿灯、路况、定位稳定性要求高。
   - 付费可能：中低，只有路线质量稳定后才会高频使用。
4. **轻户外 / 骑行用户（优先级 P2）**
   - 场景：周末徒步、骑行、郊野路线挑战。
   - 痛点：需要更长距离和路况适配，地图数据依赖更强。
   - 付费可能：待验证，建议作为后续扩展，不作为 V1 主目标。

### 不优先服务的用户
- 只关心配速、心率、训练计划的严肃跑者：他们更依赖专业训练 App。
- 只想找附近常规路线的新手：大众地图和运动 App 已能满足基础需求。
- 不愿开启定位权限的用户：核心导航链路无法成立。

## 模块2：功能架构（按版本）

### V1.0 MVP（2-4周）
目标：闭环可用（上传 -> 生图 -> 调整里程 -> 开始导航 -> 完成报告），单仓内前后端分层。

#### V1.0 产品目标
- 验证用户是否愿意为了“跑出一个形状”完成一次真实跑步，而不是只在 App 内预览路线。
- 验证组织者是否愿意把路线作为活动玩法，用于跑团、校园、品牌或健身房挑战。
- 不追求第一版覆盖所有城市道路复杂情况，优先在可控区域生成可跑路线：公园、操场周边、滨江绿道、大学城、低车流街区。

#### 功能清单
- **定位优先打底**（半自动）：启动检测定位权限；失败则提示授权并阻断“开始导航”。
- **地图配置获取**（自动）：`GET /v1/maps/config` 返回 provider 列表、能力、密钥可用状态。
- **基础模板生成路线**（半自动）：用户选择圆形、三角形、爱心、星形等模板后，前端带当前位置、目标里程和 provider 请求后端生成路线点。
- **自定义图片生成路线**（半自动）：用户选择“自定义图片/上传图片”后，前端上传 JPG/PNG，调用 `POST /v1/routes/from-image`，由后端返回 `routeId`、`points`、路线元数据和风险信息。
- **路线预览与风险确认**（半自动）：模板或图片生成路线后，必须先进入路线预览页，展示点位、距离、起点适配、可跑性风险和图形相似度；用户确认后才允许开始导航。
- **可跑性检查**（半自动）：V1 先输出风险提示和人工确认；后续接入路网后自动避开不可通行区域。高风险场景必须阻断导航。
- **路线里程缩放**（自动）：`POST /v1/routes/{routeId}/adjust`，保留形状并做角度平滑。
- **起点/终点重映射**（半自动）：`POST /v1/routes/{routeId}/rebase`，不重识别即可重锚定。
- **导航会话**（自动）：`start-run` -> `location` 上报 -> `state` 查询 -> `finish`。
- **偏离提示**（自动）：距离阈值分级提示，弱网时给出“本地导航降级”文案。
- **历史列表**（自动）：`GET /v1/users/me/runs`。

> 手动项：图片上传、模板选择、是否编辑起点终点、是否接受风险提示、是否触发开始导航。
> 自动项：坐标转换、路径缩放、会话进度计算、偏离评估、风险分级建议。

### V2.0 增强（1-2个月）
- AI 边缘识别升级（先行：OpenCV/ML Kit/ONNX 分支，后续自研模型）。
- 路网约束路线生成：避开河流、封闭园区、快速路、高架、不可步行道路，优先贴合人行道/公园路/绿道。
- 历史/收藏路线库、重复生成优化。
- 分享卡片（运动数据摘要），不显示 GPX。
- 地图 provider 的 runtime 切换和 provider 限流策略。

### V3.0 社区版（3-6个月）
- 赛事/挑战模板（距离、时间、路线标签）。
- 排行榜与社交组队。
- 报名合作方赛事导入、路线挑战打卡审核。
- 商店会员体系（Pro/Creator）与创作激励。

### 产品验证指标
- **生成到开跑转化率**：生成路线后点击“开始导航”的比例，种子期目标 >= 25%。
- **开跑到完成率**：开始导航后完成并生成报告的比例，种子期目标 >= 50%。
- **路线可跑评分**：用户完成后对“是否安全、是否绕路、是否像原图”打分，5 分制目标均值 >= 3.8。
- **分享率**：完成报告后分享或保存图片的比例，种子期目标 >= 30%。
- **活动复用率**：同一个组织者是否发起第二次挑战，验证 B 端/社群价值。

### 冷启动策略
1. 先找 5-10 个跑团/校园社团试点，每个试点做 1 条定制路线。
2. 用“爱心跑、校徽跑、Logo 跑、生日数字跑”等高传播模板，不先追求任意图片都完美。
3. 每次活动收集 3 类反馈：路线是否可跑、图形是否像、是否愿意分享。
4. 种子期优先人工辅助修正路线，积累真实路线样本，再反推自动算法。

## 模块3：技术选型

### 1）前端
- **默认：React Native（Expo 或裸 RN）**
- 原因：
  - 一次开发覆盖 iOS/Android；
  - 可先接入 `react-native-maps` + 原生插件化 provider；
  - 与现有 Node.js 接口生态兼容，闭环调用简单。
- 说明：当前仓内保留 web demo 页面用于本地验收，后续替换为 RN App。

### 2）地图 SDK（定位驱动 + 多 provider 抽象）
- 先用：**高德（国内默认） + Google（国际兜底）**
- 预留：**百度/腾讯（配置开关 + stub）**
- 成本与策略：
  - 国内场景先走 AMap：更贴近国内坐标体系，偏差更小，V1 已用于服务端步行规划风险抽样。
  - 百度 Key 已配置，作为后续路网校验 fallback，不在 V1 首选路径中暴露给前端。
  - 国际场景走 Google：覆盖海外、导航体验更统一。
  - 后端统一返回 providerHint 与 crsHint，前端只做渲染。

### 3）轨迹识别算法
- **简单版（V1）**：基于图像轮廓提取 + 多边形抽样（OpenCV/像素连通/骨架化）；先落“可用闭环 demo”。
- **AI 版（V2）**：
  - 输入：图片噪声/涂鸦/文字去噪 + 分割。
  - 输出：向量化曲线 + 关键点序列。
  - 目标：识别率、起点-终点稳定性、尖角抑制。
- **路线约束层（V2 必做）**：
  - 输入：图片生成的理想几何路径 + 起点 + 目标里程。
  - 处理：将理想路径吸附到可步行/可跑道路网络，避开禁行、封闭、涉水、高架和快速路。
  - 输出：可跑路线、风险段标记、图形相似度评分、预计红绿灯/转向数量。
  - 原则：宁可降低图形相似度，也不生成明显不可跑或高风险路线。

### 4）后端
- **Node.js + Express** 保留，原因：快速实现 API 契约、便于接入已有服务。
- 持久化：
  - V1：文件态持久化（已实现 state.json，避免服务重启丢失）。
  - 后续 30 天内替换：Redis（会话） + PostgreSQL（route/run）。
- 不再直接读/写前端文件；所有地图逻辑与会话在后端集中。

## 模块4：用户操作流程

```mermaid
flowchart TD
  A[打开 App] --> B[请求定位权限]
  B -->|拒绝| C[阻止开始导航 + 引导授权/手动起点]
  B -->|同意| D[读取当前位置 + 定位精度]
  D --> E[获取地图配置 /v1/maps/config]
  E --> F{选择路线来源}
  F -->|基础模板| G[选择图形模板 + 目标里程]
  F -->|自定义图片| H[选择 JPG/PNG 并上传]

  M -->|低风险| N[提示风险并允许确认]
  M -->|中风险| O[弹窗: 调整起点/重新生成/仍然使用]
  M -->|高风险| P[阻断导航: 只能调整起点或重新生成]
  N --> Q[用户确认路线]
  O --> Q
  O --> R[POST /adjust 或 /rebase]
  P --> R
  R --> L
  Q --> S[开始导航 POST /start-run]
  S --> T[实时上报位置 POST /run/{sessionId}/location]
  T --> U[轮询状态 GET /run/{sessionId}/state]
  U --> V[偏离提示+导向动作]
  V --> W[完成 POST /run/{sessionId}/finish]
  W --> X[回显闭环报告（不导出 GPX）]
```

**关键转化点**：
1) 定位授权成功（决定是否可生成现实路线）；2) 模板/图片生成路线成功；3) 路线预览风险确认通过；4) 开始导航触发。

### 路线预览与风险确认规则

#### 触发时机
- 用户选择基础模板后，后端生成路线点成功时触发。
- 用户上传自定义图片并生成路线点成功时触发。
- 用户调整目标距离、重设起点/终点或重新生成后再次触发。

#### 检查项
| 检查项 | 判断口径 | 处理 |
| --- | --- | --- |
| GPS 精度差 | `currentAccuracy > 50m` 或无定位点 | 提醒重试定位或手动选起点 |
| 起点不匹配 | 用户当前位置离路线起点超过 100m | 推荐调整起点，不直接开始导航 |
| 路线不可通行 | 穿越水域、封闭区、快速路、高架、施工区 | 高风险，阻断导航 |
| 距离偏差大 | 目标距离与生成距离偏差超过阈值 | 推荐调整目标距离或重新生成 |
| 图形相似度低 | `shapeSimilarityScore` 低于阈值 | 提醒用户可接受或重新生成 |

#### 风险分级
- **低风险**：展示提示，不阻断；用户确认后可导航。
- **中风险**：弹窗提示，默认推荐“调整起点”或“重新生成”，允许用户选择“仍然使用”。
- **高风险**：阻断开始导航，只允许“调整起点”或“重新生成”。

#### 弹窗建议文案
- 标题：`当前起点可能不适合这条路线`
- 内容：`检测到路线附近存在不可通行区域，或当前位置定位精度较低。建议先预览并调整起点。`
- 操作：`调整起点` / `重新生成` / `仍然使用`

## 模块5：关键算法

### 5.1 图片边缘到经纬度转换
1. 图像预处理：灰度化 -> 去噪 -> Canny/阈值分割。
2. 轮廓提取：找到最大/可疑闭合轮廓，去除小噪点。
3. 骨架化：边缘中心化为单像素路径。
4. 坐标采样：按弧长重采样，得到统一长度序列。
5. 投影到地理坐标：
   - 以起点为锚点，使用“像素比例尺”映射；
   - 先生成内部 WGS84 路径，再由 provider 映射到 GCJ-02/BD-09。

示例伪代码：
```js
function imageToPath(image, startPoint, provider){
  img = preprocess(image)
  contour = extractMainContour(img)
  skeleton = thin(contour)
  points2D = resampleByArcLength(skeleton, targetCount)
  pathWgs84 = points2D.map((p)=>projectToWgs84(p, startPoint, origin2D, scale))
  return convertPath(pathWgs84, 'wgs84', providerToCrs(provider))
}
```

### 5.2 按公里数缩放但保形
```js
function scalePathPreserveShape(points, targetMeters, options){
  L0 = pathLength(points)
  k = targetMeters / L0
  center = centroid(points)
  scaled = points.map(p => {
    return {
      lat: center.lat + (p.lat - center.lat) * k,
      lng: center.lng + (p.lng - center.lng) * k
    }
  })
  scaled = limitPointCount(scaled, options.minPoints, options.maxPoints)
  return angleSmooth(scaled)
}
```
- 结果保持路径整体形态；
- 通过 `min/max waypoint` 避免抖动/过稀疏；
- 与 V1 现有 `angleSmooth` 和 `resampleByDistance` 对齐。

### 5.3 城市可跑性约束
V1 可以先做“风险提示 + 人工确认”，但不能把任意几何线直接当成真实可跑路线。城市道路限制会直接决定产品能否留存。

#### 约束优先级
1. **安全优先**：避开快速路、高架、隧道、机动车专用道、水域、铁路、封闭施工区域。
2. **可通行优先**：优先选择公园步道、绿道、人行道、低车流街区、校园开放道路。
3. **连续性优先**：减少复杂掉头、跨越主干道、频繁穿越路口。
4. **形状相似度最后优化**：在安全和可通行基础上，尽量贴近原图。

#### 路线质量输出
- `runnableScore`：可跑评分，0-100。
- `shapeSimilarityScore`：图形相似度，0-100。
- `riskSegments`：风险路段列表，包含类型、坐标范围、建议处理。
- `manualReviewRequired`：是否需要用户或活动组织者确认。
- `riskLevel`：风险等级，`low` / `medium` / `high`。
- `startPointStatus`：起点适配状态，包含当前位置到路线起点距离、定位精度、是否建议重设起点。
- `confirmRequired`：是否必须弹窗确认后才能继续。

#### V1 降级方案
- 没有路网数据时，只允许在用户确认区域内生成路线，并展示“请确认路线不穿越封闭道路、水域或机动车专用路”的确认提示。
- 对复杂图片降低承诺：优先推荐基础图形、文字缩写、Logo 简化版，不承诺任意图片 100% 还原。
- V1 风险检查基于定位精度、起点距离、路线点异常跨度、距离偏差做轻量判断；当 provider 为 `amap` 且服务端配置 `AMAP_KEY` 时，额外调用高德步行规划对路线片段做抽样绕行判断。
- 百度地图 Key 作为后续 fallback：当高德不可用或路线在百度数据覆盖更稳定时，再扩展百度步行路线规划校验。

### 5.4 模板与自定义图片的生成边界
- **基础模板**：圆形、三角形、爱心、星形等不应只在前端画静态图；进入导航前仍要通过后端生成实际路线点，方便统一保存、调整、重映射和会话启动。
- **自定义图片**：用户选择本地图片后必须上传给后端，由后端识别轮廓并返回路线点；前端只负责预览、调整和确认。
- **手动编辑**：属于独立能力，后续可作为识别失败后的降级选项，不等同于“自定义图片”。
- **统一出口**：无论来源是模板还是图片，最终都落到同一个路线预览确认页，再进入导航。

## 模块6：导航体验
- **偏离提示**：每 2 秒计算 `最新定位点 -> 路线最近点距离 d`。
  - d < 10m：正常
  - 10-25m：轻提示
  - >25m：重提示 + 给予“回撤到最近路点”指引
- **路径比对**：持续比较 `actualPath` 与 `route.points`，输出 `deviationM / progressPct / currentAccuracy`。
- **弱网策略**：
  - 无法上报时本地缓冲位置，提示“定位更新中”；
  - 无法取图层时退化为“离线提示模式”（文字方向 + 距离目标）；
  - 结束时补算完成率，确保不丢报告。

## 模块8：风险与应对
1. **定位精度不稳定**：预置 provider 兜底+偏离阈值动态化，增加“手动修正偏航点”。
2. **识别率不达标**：AI 识别不触发时降级为手工关键点确认。
3. **多 provider 坐标漂移**：统一内部 WGS84，后端统一转换，session 内返回 CRS 版本号。
4. **商店审核风险（隐私）**：定位文案、后台定位时机、数据保留期写入设置页并在首次授权页说明。
5. **城市路线不可跑**：V1 做风险提示和人工确认；V2 必须接入路网约束，否则不扩大城市泛用户投放。
6. **用户只尝鲜不留存**：种子期用活动挑战、模板更新、排行榜和分享卡片提高复用，不把个人自然留存作为唯一指标。
7. **图形相似度和安全冲突**：产品规则明确安全优先，必要时降低图形还原度，并给用户展示差异原因。

## 模块10：24小时第一步
